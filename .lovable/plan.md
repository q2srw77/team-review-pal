## Goal

Currently `auto-close-overdue-requests` flips any overdue, not-completed review request straight to `completed` (with `closed_reason = deadline_reached`), generates a PDF, and emails the submitter the "all complete" notification. With the new Correction step in the workflow, auto-closing to `completed` bypasses the submitter's accept/reject pass on reviewer notes.

New rule: **Auto-close should only push a stalled review forward to `correction`. Only the submitter (via `finalize-review-request`) can move it to `completed`.** This guarantees a review can't stay locked in `pending`/`in_review` forever when a reviewer never finishes, while preserving the submitter's authority over the final outcome.

## Behavior change

For each overdue request returned by the cron (today >= `complete_by`, status != `completed`):

| Current status                | New auto-close action                                                                                                                                              |
|-------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `pending` or `in_review`      | Force any non-`completed` reviewer rows to `completed`, then set request `status = 'correction'`, `closed_reason = 'deadline_reached'`. Notify submitter.          |
| `correction`                  | Skip. The submitter already owns it. (Optionally log "still in correction past deadline" â€” out of scope unless requested.)                                         |
| `completed`                   | Already excluded by the existing `.neq("status", "completed")` filter.                                                                                              |

No PDF is generated by auto-close anymore â€” PDF generation only happens at submitter finalize.

## Implementation

### 1. `supabase/functions/auto-close-overdue-requests/index.ts`

- Change the fetch filter to only target `pending` / `in_review` rows: `.in("status", ["pending", "in_review"])` (drop `correction` from scope, keep the `complete_by <= today` filter).
- Replace the current `update({ status: "completed", closed_reason: "deadline_reached" })` block with:
  1. Update all `review_statuses` rows where `request_id = r.id` and `status != 'completed'` â†’ `status = 'completed'`. The existing `auto_update_request_status` trigger then flips the request to `correction`.
  2. As a safety net (in case there were zero reviewer rows), if request status is still not `correction`, do an explicit `update({ status: "correction" })`.
  3. Set `closed_reason = 'deadline_reached'` on the request so audit trail/UI can still show "auto-advanced due to deadline".
- Remove the call to `generate-review-report`.
- Replace the `review-all-complete` email with a new submitter notification (template name `review-auto-advanced-to-correction`) carrying `{ title, platform, teamName, completedCount, totalCount, completeBy }`. (See template work below.)
- Audit log: change `action: "auto_closed"` to `action: "auto_advanced_to_correction"` with `details: { reason: "deadline_reached", forced_completed_reviewers: <count>, completed, total, complete_by }`.

### 2. New transactional email template

- Add `supabase/functions/_shared/transactional-email-templates/review-auto-advanced-to-correction.tsx` modeled on `review-all-complete.tsx`. Subject: "Review past deadline â€” your turn to review comments". Body explains the deadline passed, any incomplete reviewers were auto-finalized, and prompts the submitter to open the request, accept/reject comments, then either re-submit or mark complete.
- Register it in `supabase/functions/_shared/transactional-email-templates/registry.ts`.

### 3. `finalize-review-request` (no functional change, just confirm)

- Already requires `status = 'correction'` and `submitted_by === userId`. That stays the single path to `completed`. No edits needed.

## Out of scope

- No DB schema changes. `closed_reason` already exists; `request_status` enum already includes `correction`.
- No RLS changes. Service role already manages everything auto-close touches.
- No UI changes to `RequestDetail.tsx` â€” the existing Correction UI already handles this state for the submitter.
- No change to manual finalize flow, resubmit flow, reviewer reminders, or PDF generation.
- Behavior for requests stuck in `correction` past deadline: leave as-is for now. Can add a separate "submitter nudge" reminder later if desired.
